# Admin Auth Design

## Goal

Replace the host dashboard's raw `admin_token` flow with a single-admin username/password login, optional TOTP 2FA, and session tokens that can also be used by the desktop app.

## Scope

- `mcmon-host` owns admin credentials, sessions, and 2FA state.
- `mcmon` desktop app logs into `mcmon-host` with host URL, username, password, and optional TOTP code, then stores the returned session token.
- `mcmon-agent` authentication remains unchanged. Agent tokens and install tokens are still generated by host and embedded in generated install commands.

## Host Model

Add two SQLite-backed concepts:

- `admin_auth`: singleton row with username, password hash, optional TOTP secret, and timestamps.
- `admin_sessions`: random session token, expiry timestamp, user agent, and remote address.

On startup, if the host has no admin row, create username `admin` with a random password and print it once to the terminal logs. Existing deployments that still have `admin_token` can keep it as a migration fallback for this implementation, but the dashboard and app should use login sessions.

Passwords use bcrypt. TOTP uses `github.com/pquerna/otp/totp`.

## Host API

- `POST /api/auth/login` accepts `username`, `password`, and optional `totp_code`; returns `session_token`, `expires_at`, `username`, and `totp_enabled`.
- `POST /api/auth/logout` deletes the current session.
- `GET /api/auth/me` returns current admin identity and 2FA state.
- `POST /api/auth/password` changes username/password after validating the current password, then deletes existing sessions except the current one.
- `GET /api/auth/2fa/setup` returns a newly generated TOTP secret and provisioning URL.
- `POST /api/auth/2fa/enable` validates the setup secret with a code, then stores it.
- `POST /api/auth/2fa/disable` validates current password or current TOTP code, then clears the secret.

Admin API authorization accepts either:

- HttpOnly `mcmon_session` cookie.
- `Authorization: Bearer <session_token>`.
- Existing `admin_token` as a compatibility fallback while the transition is in progress.

## Web UI

Dashboard pages show a compact login form instead of the admin token input. After login, requests use the session cookie automatically. A logout button replaces the save-token action.

2FA setup can be text-first for this iteration: show the secret and provisioning URI, plus an input to confirm the current TOTP code. QR rendering can be added later without changing the auth model.

## Desktop App Integration

Remote settings change from `Host URL + Admin token` to `Host URL + Username + Password + optional 2FA code`. The app calls `/api/auth/login`, stores the returned session token in local config, and proxies remote API requests using `Authorization: Bearer <session_token>`.

The app keeps local monitoring fully independent. If no remote host is configured or login fails, local monitoring continues to work.

## Security Notes

- Session tokens are random 32-byte hex strings and expire after 30 days.
- Login responses are intentionally generic for bad username/password.
- TOTP code failures do not reveal whether the password was correct beyond requiring 2FA only after password validation.
- Session cookie is HttpOnly, SameSite=Lax, and Secure when the request is HTTPS.
- Existing agent auth remains separate from admin auth.
